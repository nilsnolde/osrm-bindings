# osrm-bindings

Python bindings for [osrm-backend](https://github.com/Project-OSRM/osrm-backend) using [nanobind](https://github.com/wjakob/nanobind), built with [scikit-build-core](https://github.com/scikit-build/scikit-build-core).

## Build System

**Stack**: scikit-build-core + nanobind + CMake (3.28+) + FetchContent

The build has two paths:
1. **System OSRM found** (`FindLibOSRM.cmake`): links against pre-installed libosrm
2. **No system OSRM** (default): FetchContent clones from `OSRM_GIT_REPO` at `OSRM_GIT_TAG` (both set in `pyproject.toml`)

The FetchContent path uses `EXCLUDE_FROM_ALL` to suppress OSRM's install rules (prevents .lib files, headers, pkgconfig from polluting the wheel). OSRM executables are explicitly depended on via `add_dependencies()` and installed separately.

### Building locally

```bash
pip install -e ".[dev]"    # editable install with dev deps
pip install .              # regular install
```

### Key CMake variables (set via pyproject.toml)

- `OSRM_GIT_REPO` — OSRM repository URL (currently nilsnolde fork with upstream PRs)
- `OSRM_GIT_TAG` — commit/tag to fetch
- `ENABLE_CONAN` — Windows only, enables conan for dependencies
- `CMAKE_BUILD_TYPE` — always Release
- `CMAKE_CXX_SCAN_FOR_MODULES` — OFF (ccache doesn't support GCC module scanning)

### Patches

Only one patch is applied via `PATCH_COMMAND`:
- `patches/conan-static-deps.patch` — removes `shared=True` from bzip2 and xz-utils in OSRM's conanfile.py so wheels are self-contained. hwloc stays shared (required by onetbb).

Other fixes (conan source dir, sol2 lua compat) are in the OSRM fork.

### Wheels

- Stable ABI: `cp312-abi3` — one wheel per platform, works on Python 3.12+
- Platforms: Linux (manylinux_2_28 x86_64), macOS (arm64), Windows (amd64)
- Version: managed by setuptools-scm from git tags

## C++ Bindings

Module entry point: `src/osrm_nb.cpp` defines `NB_MODULE(osrm_ext, m)`.

### Structure

```
src/
  osrm_nb.cpp              # Module definition, OSRM class + service methods (Route, Match, etc.)
  engineconfig_nb.cpp       # EngineConfig binding
  parameters/*_nb.cpp       # One file per parameter class (Route, Match, Nearest, Table, Tile, Trip)
  types/*_nb.cpp            # Coordinate, Bearing, Approach, JSONContainer, Optional
  utility/
    osrm_utility.cpp        # kwargs processing, status checking
    param_utility.cpp        # assign_baseparameters(), assign_routeparameters(), enum helpers
include/                    # Headers mirror src/ structure
```

### Patterns

- **Enum binding**: string-to-enum via `osrm_nb_util::str_to_enum()` with static `unordered_map`. Each enum gets `__init__(str)` + `__repr__()` + `nb::implicitly_convertible<std::string, EnumType>()`.
- **Parameter constructors**: kwargs constructor calls `assign_baseparameters()` / `assign_routeparameters()` helpers in `param_utility.cpp`.
- **Hints**: exposed as `list[str | None]` (base64 strings), converted to/from `osrm::engine::Hint` via `FromBase64()`/`ToBase64()` in the binding layer.
- **Coordinate lon/lat**: exposed as `float` (degrees) via `def_prop_rw` lambdas that convert from OSRM's internal `FixedLongitude`/`FixedLatitude`.
- **JSON results**: `Object` and `Array` wrap OSRM's json types. `__getitem__` returns `nb::object` (not the raw variant) so stubs stay clean.

### Type stubs

`src/osrm/osrm_ext.pyi` is auto-generated by `nanobind_add_stub()` in CMakeLists.txt. It's committed so mkdocstrings can build docs without compiling the extension. CI verifies it's up to date (check_stubs job).

When changing C++ bindings:
1. Rebuild: `pip install -e .`
2. Commit the updated `.pyi` file

Every `.cpp` file that returns `std::string` from lambdas must `#include <nanobind/stl/string.h>`, otherwise stubs show mangled C++ type names.

## CI/CD

### Workflows

| Workflow | Trigger | What it does |
|----------|---------|--------------|
| `pull_request.yml` | PRs to main, workflow_dispatch | Lint + build all platforms + check stubs |
| `push_master.yml` | Push to main | Build all platforms |
| `gh-pages.yml` | Push to main | Build and deploy docs (no compilation needed) |
| `publish_wheels.yml` | Tags, workflow_dispatch | Build + publish to TestPyPI (trusted publisher) |

### Platform-specific build details

| | Linux | macOS | Windows |
|---|---|---|---|
| Container | manylinux_2_28 (custom image) | none | none |
| Compiler cache | ccache (pre-installed in image) | ccache (brew) | sccache (pip) |
| Cache launcher | Set in image, NOT via CMAKE_*_LAUNCHER | `CIBW_CONFIG_SETTINGS_MACOS` | `CIBW_CONFIG_SETTINGS_WINDOWS` |
| OSRM deps | Baked into manylinux image | brew (lua, tbb, boost) | Conan (`ENABLE_CONAN=ON`) |
| Wheel repair | auditwheel | delocate (default) | delvewheel |

**Important**: `CIBW_CONFIG_SETTINGS_*` env vars **replace** (not merge with) `config-settings` in pyproject.toml. So Windows must include `ENABLE_CONAN=ON` in the GHA env var, not just in pyproject.toml.

### Caching strategy

- **ccache/sccache**: keyed on `OS-branch-run_id`, restored by prefix match (latest from same branch, then any branch)
- **Conan**: keyed on `OS-hash(pyproject.toml)` since OSRM_GIT_TAG determines which conanfile is used. Uses `!cancelled()` on save to avoid caching partial downloads from cancelled runs.

### workflow_dispatch options

- `pull_request.yml` has `test-windows` boolean input (default true) — can be set to false to skip slow/flaky Windows tests

## Testing

```bash
# Build test data (requires installed osrm executables)
cd tests/data && make

# Load shared memory datastore
python -m osrm datastore tests/data/ch/monaco

# Run tests
pytest tests/
```

Tests cover all OSRM services: route, match, nearest, table, tile, trip. Test data is Monaco OSM extract preprocessed for CH and MLD algorithms.

## Documentation

MkDocs Material + mkdocstrings. Reads type stubs from `src/osrm/` via `paths: ["../src"]` in mkdocs.yml. No compilation needed.

```bash
cd docs && pip install -r requirements.txt && mkdocs serve
```

## Documentation Maintenance

After changes to build system (`pyproject.toml`, `CMakeLists.txt`), CI workflows, platform requirements, test setup, or release process, check whether `docs/docs/pages/development.md` needs updating.
